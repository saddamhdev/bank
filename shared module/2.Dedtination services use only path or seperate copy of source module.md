In a **multi-module Maven project**, the destination service does **NOT** use a path to the source folder and does **NOT** keep a separate copy of the source module.

It uses the **compiled JAR** of that module automatically.

---

## ✅ How It Works

Example structure:

```text
parent-project
 ├── common-lib
 ├── user-service
 └── order-service
```

If `order-service` needs classes from `common-lib`, you add:

```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>common-lib</artifactId>
    <version>1.0.0</version>
</dependency>
```

### What happens?

1. Maven builds `common-lib`
2. Generates `common-lib-1.0.0.jar`
3. `order-service` uses that JAR in its classpath

✔ No manual path
✔ No copying source code
✔ No duplicate source module

---

## ❌ What You SHOULD NOT Do

You should NOT:

* Import source folder manually
* Copy classes into another module
* Use file path like `../common-lib/src/...`

That breaks proper module design.

---

## 🧠 Important: Development vs Runtime

| Stage       | What is Used          |
| ----------- | --------------------- |
| Development | Source visible in IDE |
| Build Time  | Compiled JAR          |
| Runtime     | Compiled JAR          |

Your IDE shows source because everything is in the same project — but technically dependency works through JAR.

---

## 🔥 Special Case

If modules are in **different repositories**, then:

* You must `mvn install` first
* Or publish to Nexus
* Then other project downloads the JAR

Still no source copy — only JAR dependency.

---

### Final Clear Answer:

👉 Destination service uses **compiled JAR automatically managed by Maven**
👉 It does NOT use source path
👉 It does NOT create separate copy of source
